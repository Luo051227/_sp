## 程式一：銀行存款模擬

### 1.1 問題描述

模擬同一個銀行帳戶的多執行緒存提款操作：
- 初始存款：0 元
- 4 個執行緒執行存款
- 4 個執行緒執行提款
- 每個執行緒執行 100,000 次
- 每次金額：100 元

### 1.2 預期結果

由於存款次數和提款次數相同（各 400,000 次），且每次金額相同，最終餘額應為 0 元。

### 1.3 程式架構

```c
typedef struct {
    int balance;              // 存款餘額
    pthread_mutex_t mutex;   // 互斥鎖
} Account;
```

### 1.4 同步機制

使用 **pthread_mutex** 保護共享的帳戶餘額：

```c
void* deposit_thread(void* arg) {
    for (int i = 0; i < DEPOSIT_COUNT; i++) {
        pthread_mutex_lock(&account.mutex);  // 加鎖
        account.balance += AMOUNT;           // 臨界區
        pthread_mutex_unlock(&account.mutex); // 解鎖
    }
}
```

### 1.5 競爭條件事說明

如果沒有互斥鎖，可能發生以下情況：

```
時間線：
T1: 執行緒A讀取餘額 (0)
T2: 執行緒B讀取餘額 (0)
T3: 執行緒A計算並寫入 (100)
T4: 執行緒B計算並寫入 (100)  <- 覆蓋了T3的結果！

結果：餘額 = 100（正確應該是 200）
```

### 1.6 編譯與執行

```bash
gcc -pthread -o bank 銀行存款模擬.c
./bank
```

## 程式二：生產者-消費者問題

### 2.1 問題描述

經典的執行緒同步問題：
- 緩衝區大小：10
- 2 個生產者，每個生產 100 個產品
- 2 個消費者，每個消費 100 個產品
- 生產者將產品放入緩衝區
- 消費者從緩衝區取出產品

### 2.2 同步機制

使用兩種同步機制：

1. **訊號量（Semaphore）**
   - `empty_slots`：空位數量，初始為緩衝區大小
   - `full_slots`：產品數量，初始為 0

2. **互斥鎖（Mutex）**
   - 保護緩衝區的實際存取

### 2.3 演算法流程

```
生產者：
  while (有產品要生產) {
      wait(empty_slots);    // 等待空位
      lock(mutex);          // 進入臨界區
      放入產品到緩衝區
      unlock(mutex);        // 離開臨界區
      signal(full_slots);   // 增加產品數量
  }

消費者：
  while (有產品要消費) {
      wait(full_slots);     // 等待產品
      lock(mutex);          // 進入臨界區
      從緩衝區取出產品
      unlock(mutex);        // 離開臨界區
      signal(empty_slots);  // 增加空位數量
  }
```

### 2.4 環形緩衝區

使用環形緩衝區實現 FIFO：

```c
typedef struct {
    int buffer[BUFFER_SIZE];
    int in;      // 下一個放入位置
    int out;     // 下一個取出位置
    int count;   // 當前產品數量
} Buffer;

// 放入
buffer[in] = item;
in = (in + 1) % BUFFER_SIZE;

// 取出
item = buffer[out];
out = (out + 1) % BUFFER_SIZE;
```

### 2.5 編譯與執行

```bash
gcc -pthread -o producer_consumer 生產者消費者.c
./producer_consumer
```

## 程式三：哲學家用餐問題

### 3.1 問題描述

經典的死結問題：
- 5 位哲學家圍坐在圓桌旁
- 每位哲學家左右各有一支叉子
- 只有拿到兩支叉子才能吃麵
- 吃完後放下叉子思考

### 3.2 問題分析

**可能發生的死結場景：**
```
所有哲學家同時拿起左邊的叉子
→ 每位哲學家都在等待右邊的叉子
→ 形成循環等待 → 死結
```

### 3.3 解決策略

本程式實現了**奇偶策略**來避免死結：

```c
// 奇數號哲學家：先拿左邊，再拿右邊
// 偶數號哲學家：先拿右邊，再拿左邊

if (id % 2 == 1) {  // 奇數
    first = left_fork;
    second = right_fork;
} else {            // 偶數
    first = right_fork;
    second = left_fork;
}
```

**為何有效：**
- 打破了循環等待的條件
- 至少有一位哲學家不會與鄰居形成等待環

### 3.4 程式流程

```
哲學家迴圈（吃 MEALS_TO_EAT 餐）：
  1. 思考 (sleep)
  2. 嘗試拿起叉子
     - 先拿第一支叉子（根據策略）
     - 再拿第二支叉子
  3. 吃飯 (sleep)
  4. 放下叉子
```

### 3.5 叉子狀態表示

```c
int forks[NUM_PHILOSOPHERS];
// 0 = 可用
// 1 = 已被佔用
```

### 3.6 編譯與執行

```bash
gcc -pthread -o philosophers 哲學家用餐.c
./philosophers
```

## 執行緒同步機制比較

| 機制 | 用途 | 特性 |
|------|------|------|
| Mutex | 保護共享資源 | 独占鎖，只有lock的執行緒能unlock |
| Semaphore | 資源計數 | 可以計數，允許多個執行緒同時訪問 |
| Condition Variable | 等待特定條件 | 需要與mutex配合使用 |
| Read-Write Lock | 讀多寫少場景 | 讀者並行，寫者独占 |

## 測試與驗證

所有程式都經過測試，確保：

1. **銀行存款模擬**
   - 最終餘額正確（無競爭條件事）
   - 所有執行緒完成任務

2. **生產者-消費者**
   - 生產數量等於消費數量
   - 緩衝區正確運作（不超限、不下溢）

3. **哲學家用餐**
   - 不會發生死結
   - 所有哲學家都能完成用餐

## 結論

這三個程式展示了多執行緒程式設計中的核心概念：
- 競爭條件事需要使用互斥鎖來防止
- 執行緒間的協調可以使用訊號量
- 死結可以透過改變資源請求順序來避免

理解這些概念對於編寫正確、高效的多執行緒程式至關重要。
