# 系統程式設計

## 第一章：系統程式概述

### 1.1 什麼是系統程式

系統程式（System Programming）是電腦科學中專門為電腦系統硬體和軟體提供服務的程式設計領域。與一般應用程式不同，系統程式直接與硬體互動，提供作業系統、編譯器、驅動程式等基礎設施軟體。系統程式的核心目標是效率和控制，常需要處理低層次的資源管理、記憶體操作和硬體抽象等任務。

系統程式的特點包括：需要對硬體架構有深入了解、追求最高的執行效率、必須處理並發和同步問題、需要嚴格的資源管理和錯誤處理、以及強調可移植性和標準遵循。舉例來說，作業系統核心、驅動程式、編譯器和連結器都屬於系統程式的範疇。

系統程式設計培養了程式設計師對電腦系統的深層理解，這對於除錯、效能最佳化和安全研究都有極大幫助。許多高階應用程式的效能瓶頸往往需要透過系統層級的理解來解決。

### 1.2 系統程式的發展歷史

系統程式的發展與電腦硬體的演進密切相關，經歷了以下重要階段：

1950年代：組合語言是系統程式的主要工具，程式設計師需要直接操作機器碼和記憶體位址。這個時期的系統程式非常硬體相依，移植性極差。

1960年代：作業系統開始出現，如IBM的OS/360和MULTICS。作業系統提供了硬體抽象層，讓應用程式不需要直接處理硬體細節。這個時期也出現了早期的編譯器和高階語言。

1970年代：UNIX作業系統的誕生標誌著系統程式的重要里程碑，C語言的發明使得系統程式設計更加便捷且更具可移植性。POSIX標準的制定進一步推動了系統程式的標準化。

1980年代和1990年代：個人電腦的興起帶動了BIOS、驅動程式等系統軟體的發展。微軟的Windows和蘋果的Mac OS成為主流作業系統。網路功能的加入使得系統程式更加複雜。

2000年後：虛擬化技術和雲端運算進一步推動了系統程式的創新。多核處理器的普及帶來了新的並發程式設計挑戰。智慧型手機的興起帶動了嵌入式系統程式的發展。

### 1.3 系統程式的重要性

系統程式在計算機科學中扮演至關重要的角色。首先，它是硬體設備與應用軟體之間的橋樑，負責管理CPU、記憶體、磁碟和網路等資源。沒有系統程式，應用程式將無法在硬體上運行。

其次，系統程式設計培養了程式設計師對電腦系統的深層理解，這對於除錯、效能最佳化和安全研究都有極大幫助。了解系統底層運作原理的程式設計師，能夠編寫出更高效、更可靠的程式碼。

此外，系統程式是資訊安全的基礎。許多安全漏洞都與系統程式的設計缺陷有關，理解系統程式的運作機制對於發現和修補安全漏洞至關重要。

### 1.4 系統程式的分類

系統程式可分為以下幾大類：

1. 作業系統核心（Kernel）：管理硬體資源，提供程序管理、記憶體管理、檔案系統等功能。

2. 設備驅動程式（Device Drivers）：作為硬體設備與作業系統之間的介面，控制設備的運作。

3. 編譯器（Compilers）：將高階語言程式翻譯成機器碼，如GCC、Clang等。

4. 組譯器（Assemblers）：將組合語言翻譯成機器碼，如MASM、NASM等。

5. 連結器（Linkers）：將多個目標檔案合併成單一的可執行檔案，如ld、link等。

6. 載入器（Loaders）：將可執行檔案載入記憶體並開始執行。

7. 除錯器（Debuggers）：幫助程式設計師找出和修復程式錯誤，如GDB、WinDbg等。

8. 系統工具（System Utilities）：提供系統管理和維護功能，如ls、ps、top等。

每種系統程式都有其特定的功能和設計目標，彼此協作形成完整的系統軟體生態。

## 第二章：組合語言與機器碼

### 2.1 機器語言基礎

機器語言是電腦CPU直接執行的二進制指令集，是程式設計的最低層次。每種處理器架構都有其獨特的指令集架構（Instruction Set Architecture，ISA），如x86、ARM、MIPS、RISC-V等。不同的ISA有不同的指令格式、暫存器和記憶體模型，這也決定了系統程式的可移植性。

機器指令由操作碼（Opcode）和運算元（Operands）組成。操作碼指定要執行的操作，如加法、跳轉、資料傳送等。運算元指定操作涉及的資料或記憶體位置，可以是暫存器、立即數或記憶體位址。

機器語言是最低層次的程式設計語言，雖然難以閱讀和編寫，但具有最高的執行效率。在效能關鍵的場景中，組合語言或許是必要的選擇。

### 2.2 組合語言的特性

組合語言是機器語言的可讀性表示，使用助記符（Mnemonics）代替二進制操作碼。例如，x86架構中的「MOV」代表資料傳送操作，「ADD」代表加法操作，「JMP」代表無條件跳轉。

組合語言還允許使用標籤（Labels）來標記程式碼和資料的位置，使用巨集（Macros）來簡化重複的程式碼。不同的CPU架構有不同的組合語言語法，但基本概念是相似的。

組合語言程式的結構通常包括：偽指令（Directive）用於定義資料和區段；標籤（Label）用於標記位置；指令（Instruction）則是實際的CPU操作。

### 2.3 組合語言程式設計實踐

編寫組合語言程式需要了解目標架構的暫存器組、記憶體模型和指令集。以x86-64架構為例，主要暫存器包括：

- 通用暫存器：RAX、RBX、RCX、RDX、RSP、RBP、RSI、RDI
- 額外暫存器：R8、R9、R10、R11、R12、R13、R14、R15
- 特殊暫存器：RIP（指令指標）、RFLAGS（狀態標誌）

程式設計師需要手動管理堆疊，掌握呼叫約定（Calling Conventions），並理解各種記憶體定址模式，如直接定址、間接定址、基址加偏移定址等。

組合語言常用於：效能關鍵的程式碼片段、嵌入式系統開發、惡意軟體分析、BIOS和驅動程式開發、以及作業系統核心開發。

### 2.4 組譯器原理

組譯器（Assembler）是將組合語言程式翻譯成機器碼的工具。現代組譯器支援巨集處理、條件編譯和符號除錯資訊。

兩遍掃描（Two-pass Assembly）是常見的實現方式：
- 第一遍：掃描整個程式，收集所有標籤和符號的位址，建立符號表
- 第二遍：根據符號表生成機器碼，並處理前向引用

組譯器還負責解析偽指令（Pseudo-instructions），如.define（定義常數）、.byte（分配位元組）、.word（分配字）等。

常見的組譯器包括：NASM（Netwide Assembler）、MASM（Microsoft Macro Assembler）、FASM（Flat Assembler）和GAS（GNU Assembler）。

## 第三章：編譯器設計

### 3.1 編譯器的基本結構

編譯器是將高階語言程式翻譯成目標機器碼的系統軟體。它通常分為前端（Frontend）和後端（Backend）兩部分，這種分離設計使得同一個編譯器可以支援多種源語言和多個目標平台。

前端負責：
- 詞彙分析（Lexical Analysis）：將程式字元流轉換為符號流
- 語法分析（Syntax Analysis）：根據文法規則檢查程式語法
- 語意分析（Semantic Analysis）：進行型別檢查和作用域解析
- 中間代碼生成：產生與目標無關的中間表示

後端負責：
- 中間表示最佳化：進行與機器無關的最佳化
- 目標碼生成：將最佳化後的中間表示轉換為機器碼
- 程式碼配置：處理目標機器的特定優化

### 3.2 詞彙分析

詞彙分析（Lexical Analysis）是編譯的第一階段，將原始程式字元流轉換為符號流（Token Stream）。詞彙分析器（Lexer）識別以下類型的符號：

- 識別符號（Identifier）：變數名、函數名等
- 關鍵字（Keyword）：if、while、for等保留字
- 運算子（Operator）：+、-、*、/等
- 分隔符（Delimiter）：括號、分號等
- 常數（Literal）：數值、字串等

正規表示式（Regular Expressions）是描述詞彙模式的常用工具。有限狀態機（Finite State Machine）是實現詞彙分析器的有效演算法。

詞彙分析還負責處理空白字元和註解的過濾，以及記錄行號和列號以便錯誤報告。

### 3.3 語法分析

語法分析（Syntax Analysis）根據文法規則檢查符號流的語法正確性，並建構語法樹（Parse Tree）或抽象語法樹（AST）。

常見的語法分析方法包括：

1. 自上而下分析（Top-down Parsing）：從文法的起始符號開始，嘗試推導出輸入的符號串。適合LL文法。

2. 自下而上分析（Bottom-up Parsing）：從輸入的符號串出發，透過歸約逐步推到起始符號。適合LR文法。

語法分析器還負責錯誤恢復，當發現語法錯誤時嘗試繼續分析以報告更多錯誤，而不是立即放棄。

### 3.4 語意分析

語意分析（Semantic Analysis）在語法分析的基礎上進行更深層的檢查，確保程式碼的意義是正確的。主要任務包括：

1. 型別檢查（Type Checking）：確保運算元的型別兼容，如不能對字串進行加法運算。

2. 作用域解析（Scope Resolution）：確定每個識別符號的定義位置，避免作用域錯誤。

3. 符號表管理（Symbol Table Management）：儲存程式中的識別符號資訊，如名稱、型別、作用域和記憶體位置。

4. 其它檢查：常數折疊、型別轉換、介面檢查等。

型別系統（Type System）定義了語言中合法的資料型別和操作，是語意分析的基礎。

### 3.5 中間表示與最佳化

中間表示（Intermediate Representation，IR）是介於源語言和目標語言之間的程式表示。良好的IR應該是：
- 易於生成和操作
- 能表達足夠的語義資訊
- 與源語言和目標語言無關

常見的IR形式包括：三地址碼（Three-Address Code）、靜態單一賦值形式（SSA）、控制流圖（CFG）等。

編譯器最佳化在IR層面進行，常見的最佳化技術包括：
- 常數傳播（Constant Propagation）：用常數值替換常數變數
- 死碼消除（Dead Code Elimination）：移除不會執行的程式碼
- 迴圈不變量移動（Loop Invariant Code Motion）：將迴圈內不變的計算移到迴圈外
- 公共子表達式消除（Common Subexpression Elimination）：避免重複計算相同的表達式
- 指令排程（Instruction Scheduling）：優化指令順序以提高管線效率

最佳化可分為與機器無關的最佳化和與機器相關的最佳化，前者在任何目標平台上都能提升效能，後者則利用目標架構的特性。

### 3.6 目標碼生成

目標碼生成（Code Generation）將最佳化後的中間表示轉換為目標機器的機器碼。這涉及多個複雜的問題：

1. 暫存器分配（Register Allocation）：決定哪些值應該存放在暫存器中，哪些應該存放在記憶體中。圖著色演算法是解決此問題的常用方法。

2. 指令選擇（Instruction Selection）：將IR操作對應到目標機器的指令序列。需要考慮指令的功能和效能。

3. 呼叫約定處理（Calling Convention Handling）：處理函數參數傳遞、返回值和堆疊框架管理。

4. 目標碼排程（Target Code Scheduling）：優化指令順序以利用目標處理器的特性。

目標碼生成還需要處理各種目標平台的特定問題，如位址空間配置、對齊要求等。

## 第四章：作業系統核心

### 4.1 作業系統概述

作業系統（Operating System）是管理硬體資源並提供服務給應用程式的系統軟體。它的主要功能包括：

1. 程序管理（Process Management）：建立、終止、排程和控制程序
2. 記憶體管理（Memory Management）：分配和回收記憶體空間
3. 檔案系統（File System）：提供持久資料儲存的抽象
4. 設備驅動（Device Driver）：管理硬體設備
5. 安全性（Security）：保護系統資源不被未授權訪問

作業系統作為硬體和應用程式之間的中介層，提供了抽象資源（如檔案、行程和訊息）來簡化程式設計。經典的作業系統包括UNIX、Linux、Windows和macOS。

### 4.2 程序與執行緒

程序（Process）是作業系統資源分配的基本單位。每個程序擁有：
- 獨立的位址空間
- 程式碼和資料
- 開啟的檔案描述符
- 訊號處理器
- 環境變數

執行緒（Thread）是程序內的執行單位。多個執行緒共享程序的資源（位址空間、檔案等），但每個執行緒有自己的：
- 執行狀態（執行、就緒、阻塞）
- 暫存器上下文
- 堆疊
- 局部變數

行程管理涉及建立、終止、排程和同步。行程狀態包括：建立（Created）、就緒（Ready）、執行（Running）、阻塞（Blocked）和終止（Terminated）。

排程演算法決定哪個行程或執行緒獲得CPU時間，常見的有：
- 先到先服務（FCFS）
- 最短工作優先（SJF）
- 時間片輪轉（Round Robin）
- 多級反饋佇列（Multilevel Feedback Queue）
- 公平分享排程（Fair-share Scheduling）

### 4.3 記憶體管理

記憶體管理是作業系統的核心功能之一，負責分配和回收記憶體空間。主要技術包括：

1. 分頁（Paging）：將記憶體劃分為固定大小的頁面，透過頁表將虛擬位址映射到實體位址。分頁提供了記憶體保護和需求分頁功能。

2. 分段（Segmentation）：提供更大的邏輯記憶體單元，每個段有自己的意義（如程式碼段、資料段）。

3. 交換（Swapping）：將不活躍的記憶體頁面移到磁碟以擴展可用記憶體。

4. 虛擬記憶體（Virtual Memory）：結合金使用分頁和分段的記憶體管理技術，提供比實際物理記憶體更大的位址空間。

記憶體最佳化技術包括：
- 碎片整理（Defragmentation）
- 記憶體池（Memory Pool）
- 垃圾回收（Garbage Collection）

### 4.4 檔案系統

檔案系統提供持久資料儲存的抽象，是作業系統的重要組成部分。

常見的檔案系統架構包括：
- FAT系列：早期DOS和Windows使用的檔案系統
- NTFS：Windows NT的核心檔案系統
- ext4：Linux常用的檔案系統
- HFS+：蘋果macOS的檔案系統
- Btrfs：Linux的新一代檔案系統

檔案系統的核心概念包括：
- 檔案（File）：命名的資料集合
- 目錄（Directory）：組織檔案的層次結構
- inode：檔案的元資料節點
- 檔案描述符（File Descriptor）：程序開啟檔案的存取點

檔案系統的一致性和可靠性透過以下機制保證：
- 日誌（Journaling）：記錄操作以便崩潰恢復
- 檢查點（Checkpoint）：定期保存系統狀態

分散式檔案系統如NFS和CIFS支援網路共享。

### 4.5 設備驅動程式

設備驅動程式是作業系統與硬體設備之間的介面。每種設備都需要特定的驅動程式來提供標準化的介面。

驅動程式的主要職責包括：
- 設備初始化
- 資料傳輸
- 緩衝管理
- 中斷處理

設備可分為兩大類：
- 字符設備（Character Device）：逐字節傳輸資料，如鍵盤、序列埠
- 區塊設備（Block Device）：以資料塊為單位傳輸，如硬碟、SSD

驅動程式的載入方式：
- 內建於核心（Integrated）
- 作為模組載入（Loadable Module）

現代作業系統提供硬體抽象層（HAL）來簡化跨平台驅動開發。

### 4.6 核心同步與並發

核心程式碼必須處理並發存取共享資源的問題。常用的同步機制包括：

1. 自旋鎖（Spin Lock）：
   - 適用於短臨界區段
   - 在鎖可用前不斷輪詢
   - 避免了上下文切換的開銷

2. 互斥鎖（Mutex）：
   - 適用於較長的臨界區段
   - 不可用時會阻塞執行緒
   - 支援遞迴鎖和優先順序繼承

3. 訊號量（Semaphore）：
   - 可用於資源計數和同步
   - 支援計數訊號量和二進制訊號量

4. 條件變數（Condition Variables）：
   - 支援等待特定條件
   - 必須與互斥鎖配合使用

優先順序反轉問題需要特殊處理，常用方案包括：
- 優先順序繼承（Priority Inheritance）
- 優先順序天花板（Priority Ceiling）

## 第五章：連結器與載入器

### 5.1 目標檔案格式

目標檔案（Object File）是編譯器生成的機器碼和中繼資料的容器。常見的目標檔案格式包括：

1. ELF（Executable and Linkable Format）：Unix和Linux系統的標準格式
2. PE（Portable Executable）：Windows系統的格式
3. Mach-O：macOS系統的格式

ELF格式包含：
- ELF標頭（Header）：Overall的元資料
- 程式標頭（Program Header）：段（Segment）的位置
- 區段（Sections）：不同類型的資料
- 區段標頭（Section Headers）：區段的描述

常見的區段類型包括：
- .text：程式碼
- .data：已初始化資料
- .bss：未初始化資料
- .rodata：唯讀資料
- .symtab：符號表
- .rel/.rela：重工表

### 5.2 符號管理

符號（Symbol）代表目標檔案中的名稱，如函數和變數。符號表包含：
- 符號名稱
- 符號類型（函數、變數、標籤等）
- 符號位址
- 符號作用域

符號類型包括：
- 全域符號（Global Symbol）：可被其他檔案引用
- 區域符號（Local Symbol）：僅在當前檔案中可見
- 強符號（Strong Symbol）：定義且初始化
- 弱符號（Weak Symbol）：可能未定義

符號解析是將符號引用與符號定義匹配的過程。常見的符號衝突在連結時解決，強符號可以覆蓋弱符號。

### 5.3 重定位

重定位（Relocation）將相對位址調整為最終的執行位址。目標檔案中的位址是相對於某個基準的，載入後需要加上基址。

重工表（Relocation Table）記錄：
- 需要重工的位置
- 重工的類型
- 需要加上的符號值

常見的重工類型包括：
- R_X86_64_PC32：相對PC的32位元位址
- R_X86_64_PLT32：PLT條目的相對位址
- R_X86_64_GOTPCREL：GOT條目的相對位址
- R_X86_64_64：64位元絕對位址

動態連結涉及在執行時重工來自共享庫的符號。

### 5.4 靜態連結

靜態連結（Static Linking）在編譯時將所有目標檔案和庫整合為單一的可執行檔案。連結器（Linker）的工作包括：

1. 符號解析：解析所有未定義的符號引用
2. 重工處理：調整符號位址
3. 區段合併：將多個目標檔案的同類區段合併
4. 符號重排：優化符號排列

庫搜尋路徑決定如何查找靜態庫（.a文件）。連結時的庫順序很重要，未定義的符號需要在後面的庫中找到定義。

靜態連結的優點：
- 部署簡單，單一檔案
- 執行時不需要依賴庫
- 不可預測性較低

靜態連結的缺點：
- 可能導致重複的程式碼
- 執行檔案大小較大
- 庫更新需要重新連結

### 5.5 動態連結

動態連結（Dynamic Linking）在執行時解析符號。可執行檔案只包含符號引用，實際的符號解析在載入時或首次使用時進行。

共享庫（Shared Library）或動態連結庫（DLL）是動態連結的基礎。在Linux中是.so文件，在Windows中是.dll文件。

動態連結的優勢：
- 多個程式共享同一庫，節省記憶體
- 庫更新不需要重新編譯應用程式
- 允許動態載入和插件系統

延遲連結（Lazy Binding）推遲符號解析直到函數被調用，減少啟動時間。

動態連結器（Dynamic Linker）在執行時完成符號解析和重工。符號解析順序包括：可執行檔案本身的符號、依賴庫的符號等。

### 5.6 載入器

載入器（Loader）負責將可執行檔案載入記憶體並開始執行。

載入過程包括：
1. 讀取ELF標頭，驗證檔案格式
2. 解析程式標頭，確定需要映射的區段
3. 建立虛擬位址空間
4. 映射程式碼和資料區段
5. 設定堆疊和環境
6. 初始化動態連結器（如果需要）
7. 跳轉到程式入口點

載入器還需要處理：
- ASLR（位址空間配置隨機化）：隨機化載入位址以提高安全性
- 記憶體保護：設定可執行區域的權限
- 基址重設：處理PIE（Position Independent Executable）

動態連結器的角色類似於載入器，專門處理動態庫的連結和符號解析。

## 第六章：系統工具與程式開發

### 6.1 版本控制系統

版本控制系統（Version Control System）管理程式碼的歷史和變更，是現代軟體開發不可或缺的工具。

Git是目前最流行的分散式版本控制系統。其核心概念包括：
- 倉庫（Repository）：存放所有版本歷史的地方
- 提交（Commit）：記錄程式碼的變更
- 分支（Branch）：獨立的開發線
- 合併（Merge）：將分支的變更合併到一起

常用Git命令：
- git init：初始化新倉庫
- git add：將變更加入暫存區
- git commit：提交變更
- git push：推送到遠端
- git pull：從遠端拉取
- git checkout：切換分支
- git merge：合併分支

分支策略如Git Flow和GitHub Flow幫助團隊協作。版本控制還支持程式碼審查、回溯和並行開發。

### 6.2 自動化建置

自動化建置（Build Automation）工具簡化編譯和部署流程。

Make是經典的建置工具，使用Makefile定義建置規則：
```makefile
target: dependencies
    commands
```

CMake是跨平台的建置產生器，生成平台特定的建置系統（如Makefile或Ninja）。

Autotools提供GNU構建系統，適用於跨平台的C/C++專案。

現代語言通常有自己的套件管理工具：
- Cargo：Rust
- npm、yarn：JavaScript
- pip：Python
- Maven、Gradle：Java

持續整合（CI）工具如Jenkins、GitHub Actions和GitLab CI自動化測試和部署。

### 6.3 調試工具

調試（Debugging）是找出和修復程式錯誤的過程。

GDB（GNU Debugger）是Linux下主要的命令列調試器。常用功能包括：
- 設定中斷點（break）
- 逐步執行（step/next）
- 檢視記憶體（x命令）
- 檢視暫存器（info registers）
- 修改變數值（set）
- 分析崩潰輸出（backtrace）

Valgrind提供記憶體錯誤檢測和效能分析，常用工具包括：
- memcheck：檢測記憶體泄漏和錯誤
- callgrind：分析函數呼叫

perf是Linux的效能分析工具，可以分析熱點和事件。

靜態分析工具在不運行的情況下發現潛在問題，如Coverity、PVS-Studio、clang-tidy。

### 6.4 效能分析

效能分析（Profiling）識別程式的效能瓶頸。

熱點分析（Hot Spot Analysis）找出消耗最多時間的程式碼區段，幫助確定最佳化的重點。

呼叫圖（Call Graph）顯示函數之間的呼叫關係和時間分布，幫助理解程式的執行流程。

快取命中率分析幫助優化記憶體訪問模式，提高CPU快取利用率。

常用效能分析工具：
- gprof：GNU效能分析器
- perf：Linux效能分析工具
- Valgrind的Callgrind
- Intel VTune
- Xcode Instruments

效能分析需要代表性的工作負載和準確的測量方法。

### 6.5 測試框架

測試是確保軟體品質的重要手段。

單元測試（Unit Testing）驗證個別函數或模組的正確性。常用框架：
- Google Test（C++）
- JUnit（Java）
- pytest（Python）
- Jest（JavaScript）

整合測試（Integration Testing）驗證多個模組的協作，確保模組之間的介面正確。

系統測試（System Testing）在完整環境中測試整個應用。

測試驅動開發（TDD）強調先寫測試再寫產品代碼，確保測試覆蓋和重構安全。

覆蓋率工具測量測試對程式碼的覆蓋程度，如gcov（GCC）、coverage.py（Python）。

### 6.6 程式碼品質

靜態分析檢查程式碼的潛在問題而不執行它。

Linter工具：
- ESLint（JavaScript）
- Pylint（Python）
- clang-tidy（C/C++）
- gofmt（Go）

圈複雜度（Cyclomatic Complexity）測量程式碼的複雜程度，過高的複雜度表示需要重構。

程式碼審查（Code Review）由團隊成員人工檢查代碼，發現潛在問題和改進點。

程式碼度量：
- 行數（Lines of Code）
- 巢狀深度（Nesting Depth）
- 耦合度（Coupling）
- 內聚度（Cohesion）

品質閘門（Quality Gate）在不符合標準時阻止部署，確保交付品質。

## 第七章：系統安全基礎

### 7.1 記憶體安全

記憶體安全問題是系統安全中最常見的漏洞類型。

緩衝區溢位（Buffer Overflow）發生在寫入資料超出分配空間時。攻擊者可以利用此漏洞覆蓋相鄰的記憶體，執行任意程式碼。

堆疊溢位（Stack Overflow）利用函數返回位址的覆蓋。當函數返回時，控制的不是原本的返回位址，而是攻擊者指定的位址。

堆溢位（Heap Overflow）攻擊堆記憶體管理結構，如堆塊的header。

其它記憶體錯誤：
- 使用-after-free：使用已釋放的記憶體
- Double-free：釋放已釋放的記憶體
- 釋放後使用（Use-after-free）

防御措施包括：
- Stack Canaries：堆疊中放置探測值
- ASLR：位址空間隨機化
- DEP/NX：資料執行保護
- Safe C Library：安全的記憶體操作API
- 控制流完整性（CFI）

### 7.2 權限管理

作業系統透過權限控制來保護資源。

Unix/Linux的檔案權限：
- 擁有者權限（Owner）
- 群組權限（Group）
- 其他人權限（Others）
- 每組權限包括：讀（r）、寫（w）、執行（x）

ACL（Access Control List）提供更細粒度的控制，可以指定特定使用者或群組的權限。

基於角色的存取控制（RBAC）將權限組織為角色，簡化管理。

強制存取控制（MAC）如SELinux提供更嚴格的政策，基於安全標籤控制存取。

使用者權限最小化原則減少攻擊面，特權程式需要仔細審查以防止權限提升。

### 7.3 系統呼叫與API

系統呼叫是使用者空間程式請求作業系統服務的介面。

常見的系統呼叫：
- 檔案操作：open、read、write、close
- 行程管理：fork、exec、wait、exit
- 網路操作：socket、connect、send、recv
- 記憶體管理：mmap、brk

安全漏洞常發生在系統呼叫的參數驗證上，如路徑穿越、命令注入。

系統API的錯誤處理不當導致資源洩漏和安全問題，如未正確關閉檔案描述符。

系統呼叫追蹤工具如strace（Linux）和Process Monitor（Windows）幫助診斷問題。

### 7.4 加密與安全通訊

加密保護資料的機密性和完整性。

對稱加密：
- AES（Advanced Encryption Standard）
- ChaCha20
- 優點：速度快
- 缺點：金鑰分發困難

非對稱加密：
- RSA
- ECC（橢圓曲線密碼學）
- 用途：密鑰交換、數位簽名

雜湊演算法：
- SHA-256、SHA-3
- Bcrypt、Argon2（密碼雜湊）
- 用於：資料完整性、密碼儲存

TLS/SSL提供安全的網路通訊：
- 證書驗證
- 金鑰交換
- 加密傳輸

密碼存儲應該使用加鹽雜湊（如bcrypt、scrypt、Argon2）而非明文。

密鑰管理是加密系統安全的關鍵，包括金鑰的產生、儲存、分發和輪換。

### 7.5 安全編程實踐

輸入驗證是防止注入攻擊的第一道防線。所有外部輸入都應該被驗證和清理。

防止各類注入攻擊：
- SQL注入：使用參數化查詢或ORM
- 命令注入：避免Shell執行，使用安全的API
- XSS：輸出編碼
- 路徑穿越：驗證路徑規範化

最小特權原則減少攻擊後的損害，程式只應具有完成任務所需的最小權限。

安全更新機制確保補丁及時部署，修復已知漏洞。

錯誤處理不應該洩露敏感資訊，如內部路徑、密碼、系統詳細資訊。

安全審計和滲透測試發現潛在漏洞，定期進行安全評估。

安全開發生命週期（SDL）將安全整合到開發流程中，從設計到部署都要考慮安全。

## 第八章：進階主題

### 8.1 虛擬化技術

虛擬化（Virtualization）在一台物理機器上運行多個虛擬機器，提高資源利用率。

hypervisor是虛擬化的核心，分為：
- Type-1（原生）：直接在硬體上運行，如VMware ESXi、Xen
- Type-2（托管）：在作業系統上運行，如VirtualBox、VMware Workstation

CPU虛擬化需要硬體輔助：
- Intel VT-x
- AMD-V

記憶體虛擬化使用第二層位址轉換：
- EPT（Extended Page Tables，Intel）
- NPT（Nested Page Tables，AMD）

網路虛擬化提供虛擬網卡和交換機，如veth、bridge。

儲存虛擬化抽象化底層儲存設備，如LVM、RAID。

容器（Containers）如Docker提供輕量級虛擬化，共享主機核心，更高效。

### 8.2 雲端運算基礎

雲端運算提供按需的計算資源，改變了IT部署模式。

服務模型：
- IaaS（基礎設施即服務）：提供虛擬機器，如AWS EC2
- PaaS（平台即服務）：提供執行環境，如Heroku
- SaaS（軟體即服務）：提供完整應用，如Gmail

部署模式：
- 公有雲：第三方提供服務
- 私有雲：企業自建
- 混合雲：結合公有和私有

雲端安全涉及：
- 資料加密
- 身份驗證和存取控制
- 網路隔離
- 合規性

雲端原生應用程式設計考慮：
- 擴展性（自動擴展）
- 韌性（容錯設計）
- 可移植性（容器化）

### 8.3 即時系統

即時系統（Real-time Systems）在確定的時間內回應事件，廣泛應用於工業控制、汽車電子等領域。

即時系統分類：
- 硬即時系統：必須在嚴格的期限內完成，否則導致系統失敗
- 軟即時系統：超過期限只導致效能下降

優先順序反轉問題需要特殊處理：
- 低優先順序任務持有高優先順序任務需要的鎖
- 導致高優先順序任務被阻塞

即時作業系統（RTOS）提供：
- 確定性的排程
- 最小化的中斷延遲
- 優先順序繼承

常用RTOS：FreeRTOS、VxWorks、RT-Thread、QNX

任務設計需要考慮：
- 任務優先順序
- 中斷處理
- 共用資源保護

### 8.4 嵌入式系統

嵌入式系統是專門為特定功能設計的計算系統，廣泛應用於物聯網、消費電子、工業控制等領域。

資源受限是嵌入式開發的主要挑戰：
- 有限的記憶體（KB到MB級別）
- 有限的計算能力
- 有限的儲存空間
- 有限的電源

微控制器（MCU）是嵌入式設備的核心：
- ARM Cortex-M系列
- AVR、PIC
- RISC-V

嵌入式作業系統：
- FreeRTOS
- Zephyr
- RT-Thread
- NuttX

開發模式：
- 裸機（Bare-metal）編程：直接控制硬體
- RTOS：在作業系統上開發
- Linux：功能複雜的嵌入式系統

安全考慮：
- 硬體信任根（Root of Trust）
- 安全啟動（Secure Boot）
- 安全更新

物聯網（IoT）設備的安全問題備受關注，需要加密通訊、身份驗證和定期更新。

### 8.5 分散式系統

分散式系統由多台獨立的計算機組成，共同提供服務。互聯網服務、雲端應用都是分散式系統的應用。

通訊模型：
- RPC（Remote Procedure Call）：遠端程序呼叫
- 訊息隊列：異步訊息傳遞
- RESTful API：HTTP基礎的API

共識演算法確保多個節點的一致性：
- Paxos
- Raft
- 區塊鏈共識

分散式事務確保跨節點操作的原子性：
- 兩階段提交（2PC）
- 三階段提交（3PC）

CAP定理指出分散式系統無法同時滿足：
- 一致性（Consistency）
- 可用性（Availability）
- 分割容錯（Partition Tolerance）

設計考量：
- 負載平衡
- 容錯和冗餘
- 資料分區
- 一致性模型

### 8.6 未來發展趨勢

系統程式正朝著更高的抽象層次和更強的安全性的方向發展。

Rust語言在系統編程中獲得廣泛關注：
- 記憶體安全保證
- 無GC的效能
- 越來越多系統軟體使用Rust開發

WebAssembly提供安全的可移植執行環境：
- 瀏覽器中的高效執行
- 伺服器端和嵌入式應用

eBPF允許安全的核心擴展：
- 在作業系統核心中運行沙盒程式
- 效能監控、網路追蹤、安全監控

机密計算（Confidential Computing）使用硬體保護資料：
- Intel SGX、AMD SEV
- 雲端環境中的安全計算

人工智慧助手協助系統編程：
- 程式碼生成
- 錯誤診斷
- 效能優化建議

量子計算帶來新的系統設計挑戰：
- 後量子密碼學
- 量子作業系統

JIT（即時編譯）：執行時將位元組碼編譯為機器碼。

POSIX（可移植作業系統介面）：Unix系統的標準介面。

TLS（執行緒本地儲存）：執行緒私有的全域變數。
